# 有效链路标记说明

本文档记录插件面板中“每次请求记录”的可视化标记，用来判断哪些实现路径真正生效，后续重构时可以据此删除无效或重复路径。

## 查看位置

打开插件控制面板，进入“命中对比”页，在“每次请求记录”表格中查看：

- `记录链路`
- `usage 链路`

这两个字段是调试标记，不影响请求发送、Prompt 重排或后端返回。

## 记录链路

`记录链路` 表示请求快照是通过哪种方式创建的。

| 标记 | 含义 | 结论 |
| --- | --- | --- |
| `事件记录` | `CHAT_COMPLETION_PROMPT_READY` 事件创建了请求快照。 | SillyTavern 事件链路有效。 |
| `事件+fetch` | 事件先创建快照，fetch 又从真实 `/generate` 请求体补齐模型、来源、body hash 等字段。 | 当前最理想，说明事件链路和真实请求兜底都有效。 |
| `fetch 兜底` | 没有可复用的事件快照，插件从实际 `/api/backends/chat-completions/generate` 请求体创建快照。 | 真实请求捕获有效；如果长期只出现这个标记，说明事件记录路径可能失效或时机不稳定。 |
| `未知` | 历史旧记录或未标记记录。 | 不能用于判断链路有效性。 |

判断建议：

- 如果连续多次正常生成都显示 `事件+fetch`，可以保留事件记录作为主路径，fetch 作为校准和兜底。
- 如果连续多次只显示 `fetch 兜底`，说明实际有效的是 fetch 请求体捕获，后续可以重点保留这条路径，并重新评估事件记录路径。
- 如果没有任何记录，但后端 usage 区有数据，说明记录开关、IndexedDB 或记录创建逻辑仍有问题。

## usage 链路

`usage 链路` 表示后端 usage 是通过哪种方式捕获的。

| 标记 | 含义 | 结论 |
| --- | --- | --- |
| `JSON usage` | fetch 观察器从非流式 JSON 响应中捕获 `usage`。 | 非流式 usage 捕获有效。 |
| `流式 usage` | fetch 观察器从流式 SSE 文本中捕获最终 `usage` chunk。 | 流式 usage 捕获有效，前提是上游实际转发 usage。 |
| `ST usage 事件` | SillyTavern 原生 `CHAT_COMPLETION_RESPONSE_USAGE` 事件返回 usage。 | 当前本地 SillyTavern 版本未确认存在该事件；如果出现，说明可作为主路径。 |
| `未收到` | 本次请求快照存在，但没有收到 usage。 | 不代表缓存没命中，只代表前端没拿到 usage。 |
| `未知 usage` | 收到了 usage，但来源没有标记。 | 需要补充调用点标记。 |

判断建议：

- 如果使用非流式输出，预期应看到 `JSON usage`。
- 如果使用流式输出，预期应看到 `流式 usage`；若长期显示 `未收到`，要检查上游是否支持 `stream_options.include_usage` 并转发最后的 usage chunk。
- 如果未来 `ST usage 事件` 稳定出现，可以考虑减少 fetch 响应解析逻辑；在当前本地版本没有该事件时，不应删除 fetch usage 捕获。

## 删除无效方式的规则

不要只凭一次生成结果删除链路。建议至少用同一模型和同一后端连续测试 5 次，并覆盖：

- 诊断开启，请求记录开启。
- 非流式生成。
- 流式生成。
- 合并诊断开启但后端 `/process` 不存在的情况。

可以考虑删除或降级某条路径的条件：

- 某条链路在上述场景中从未出现有效标记。
- 删除后仍能保留请求快照、后端 usage、请求差异对比。
- 删除不会影响 Prompt 重排主流程。

当前建议：

- 保留 `CHAT_COMPLETION_PROMPT_READY`，因为它是 Prompt 重排入口。
- 保留 fetch 请求体捕获，因为它能证明最终真实 HTTP body，并能在事件记录失效时兜底。
- 保留 fetch usage 捕获，因为当前本地 SillyTavern 没确认存在稳定的 usage 事件。
