# 重排序算法说明

本文档记录当前 Prompt 重排序算法，方便后续验证哪些策略真正提升缓存命中，并删除无效逻辑。

## 是否启用

默认启用：

- `enabled: true`
- `strategy: balanced`
- `protectRichFormat: true`

运行入口是 SillyTavern 的 `CHAT_COMPLETION_PROMPT_READY` 事件。插件只修改本次请求中的 `eventData.chat`，不修改角色卡、预设、世界书或聊天记录文件。

## 核心目标

重排序目标是让更稳定的内容形成更长的请求前缀，把频繁变化的内容从前缀中移走：

1. 稳定规则、角色静态设定、格式规则、变量规则、稳定世界信息尽量前置。
2. 当前状态、动态变量值等 `dynamic_state` 放到聊天历史之后、最新用户输入之前。
3. 最新用户输入保持在末尾语义位置。
4. 聊天历史、工具消息、富格式 HTML/CSS 块、不确定语义块保持保守处理。

## 自适应稳定块识别

插件有一个低命中率触发的自适应层，默认开启：

- 连续采样 3 次请求。
- 只观察可安全移动的 system/developer 或非最新 user 块。
- 排除工具调用、assistant/tool 消息、最新用户输入、动态状态、历史标记和富格式块。
- 如果连续 3 次都出现完全相同的 message hash，并且当前后端缓存命中率仍低于 20%，就把这些块锁定为 `adaptive_stable`。
- 锁定后不再每轮重新生成计划，后续沿用同一个稳定块顺序，避免“每次请求都改位置”。

这个策略的目的不是替代规则分类，而是用真实请求观察补充规则分类：

- 三次都相同，说明它在当前聊天/预设/世界书组合下高度稳定。
- 命中率低于 20%，说明仍有必要尝试更激进地保护稳定前缀。
- 一旦锁定，以后同一聊天内不会继续变更这批块的位置。

切换聊天时，自适应样本和锁定计划会清空，重新采样。

## 分类

插件会先分析每条 message，并分为以下主要类别：

| 类别 | 说明 | 处理方式 |
| --- | --- | --- |
| `stable_rule` | 稳定规则、核心指导 | 前置 |
| `adaptive_stable` | 连续 3 次相同且低命中率触发的稳定块 | 锁定后前置 |
| `character_static` | 角色静态设定 | 前置 |
| `format_rule` | 输出格式、模板、字数、语言规则 | 前置 |
| `variable_schema` | 变量更新规则、脚本格式 | 前置，放在动态变量值之前 |
| `world_static` | 世界书/世界信息 | 前置到核心静态设定之后 |
| `unknown_stable` | 普通 system 块 | balanced 策略下前置 |
| `dynamic_state` | 当前状态、动态变量值、状态栏等 | 移到聊天历史之后、最新用户输入之前 |
| `chat_history` | 历史 user/assistant/tool 消息 | 不参与稳定前置 |
| `latest_input` | 最新用户输入 | 固定在末尾语义位置 |
| `rich_format` | HTML/CSS/前端面板/代码块 | 原位保护 |
| `unsafe_to_move` | 无法确认安全边界的内容 | 原位保护 |

## 当前顺序

当前排序大致为：

```text
保留的前置非移动内容
稳定规则 / 角色静态 / 普通 system / 格式规则 / 变量规则 / 世界静态
聊天历史
当前状态 / 动态变量值
最新用户输入
保守不移动的尾部内容
```

其中 `dynamic_state` 的目标位置是“聊天历史之后、最新用户输入之前”。如果没有识别到最新用户输入，则放到当前请求末尾。

## 为什么动态状态要在聊天历史之后

动态状态通常每轮都会变化。如果它位于角色设定、规则、世界书和聊天历史之前，会导致后续大量 token 前缀失效。

把它放到聊天历史之后，可以让以下内容更稳定：

- 角色静态设定
- 稳定规则
- 格式规则
- 变量规则
- 大部分聊天历史

最新用户输入仍然放在最后，避免破坏常规聊天语义。

## 验证方式

开启插件面板中的：

- `启用 Prompt 消息重排`
- `启用请求诊断`
- `保存请求快照`

生成后查看“命中对比”：

- `Prompt 分析器` 中 `dynamic_state` 是否显示移动。
- `Prompt 分析器` 中是否出现 `adaptive_stable`，以及诊断里是否显示“已锁定”。
- `每次请求记录` 是否出现 `事件+fetch` 或 `fetch 兜底`。
- `请求差异对比` 中第一处变化是否尽量靠后。
- 后端 usage 中 `cached_tokens` 是否随重复请求增加。

如果 `dynamic_state` 仍在聊天历史之前，需要检查该状态块是否被识别成了其他类别，或是否因为富格式/不安全边界被保护。
